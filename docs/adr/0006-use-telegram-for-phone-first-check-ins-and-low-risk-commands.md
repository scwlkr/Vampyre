# Use Telegram for phone-first check-ins and low-risk commands

Vampyre will make Telegram the phone-first surface for alerts, daily briefs, status checks, and small operational commands such as `/status`, `/pause1min`, `/pause1hour`, `/pause1day`, and `/resume`, because the Owner may need to inspect or temporarily pause work while away from the Operator Workstation.

CLI check-ins, Telegram `/status`, and Daily Briefs should render from one Check-in Summary model so they share the same facts while using different lengths and formatting. Telegram `/status` should stay compact by default for phone use: overall state, Work Pause state, Budget Mode, selected/deferred/blocked projects with reasons, Owner-needed action, and useful links. The CLI renderer should carry full operator detail, including project, Run Journal, report, validation, and scheduler-cursor detail when useful.

The first implementation should be CLI-first for testability on the Operator Workstation and `wlkrlab`. The Check-in MVP should ship before returning to `paletteWOW` runtime sync and Pinmark hands-on validation; it includes the shared Check-in Summary, detailed CLI renderer, Authorized Telegram Chat-gated `/status`, and visible Work Pause state. Scheduled Daily Brief delivery and unauthorized-attempt threshold enforcement can follow after the basic check-in path unless they are cheap while wiring Telegram.

Daily briefs should stay short and action-oriented: runtime state, budget posture, completed work, selected/deferred/blocked projects with reasons, Owner-needed reviews, next likely action, and useful links, with raw logs included only when they explain a failure or required action. Immediate alerts should interrupt only for action-needed or risk events: daemon down or repeated failure, critical or exhausted token budget, Owner-needed blockers, approvals needed before progress can continue, PRs ready for review or merge, validation failures after useful work, Work Pause start/expiry/early resume, and repeated unauthorized Telegram command attempts.

Pause commands create a timed global Work Pause that stops new scheduler-selected Active Build Agent launches and project-changing work across the Project Portfolio, while the Central Daemon keeps heartbeats, status, daily briefs, notifications, GitHub polling, and urgent blocker reporting alive. Work Pause should not interrupt an already-running Active Build Agent; the agent should finish, write its Run Journal and reports, and surface the outcome normally. Emergency cancellation is a separate later control, not part of `/pause1min`, `/pause1hour`, or `/pause1day`. Work Pause state should be persisted in SQLite on `wlkrlab` with `paused_until`, `source`, `created_at`, and optional `reason`; the scheduler reads it before selecting new project-changing work and the Check-in Summary renders it. Work Pause should also be controllable through CLI commands such as `vampyre pause 1m|1h|1d`, `vampyre resume`, and pause status, all writing the same SQLite-backed state that Telegram commands wrap. Work Pause expires automatically when its duration ends, and `/resume` only ends the current pause early.

Authorized `/status`, pause, and `/resume` commands should send a short Telegram confirmation. Pause confirmations should include the resulting pause/resume state, expiry when paused, and whether any Active Build Agent is already running and will finish. The MVP should ingest Telegram commands by polling updates from the Central Daemon and storing processed update offset or idempotency state in SQLite, rather than requiring a public webhook endpoint. Unauthorized chats still receive no useful operational detail.

For the MVP, Telegram operational commands are accepted only from the Authorized Telegram Chat configured by `TELEGRAM_CHAT_ID`; unknown chats must receive no useful operational details, and unauthorized command attempts should be logged and counted quietly until three attempts occur within ten minutes, then send one Immediate Alert and suppress repeated alerts for one hour unless the source or rate changes materially. GitHub and CLI remain the durable approval and heavier control surfaces, so Telegram commands must not approve significant work, merge PRs, expose secrets, or become a general remote shell.
