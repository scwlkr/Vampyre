# Architecture

This describes the implementation currently present in this repo.

## Stack

- TypeScript on Node.js with ES modules.
- Package manager: `pnpm@10.33.0` through `corepack`.
- Build output: `dist/`, generated by `corepack pnpm build` and ignored by git.
- Tests: Node's test runner through `tsx --test tests/**/*.test.ts`.
- No CI workflow is present in the repo.
- No web routes, HTTP server, or database schema files are present outside the SQLite SQL embedded in `src/state/operationalState.ts`.

## Runtime Shape

Vampyre is daemon-first. The operator workstation runs commands locally, but the real runtime is `wlkrlab`.

Default runtime paths:

```txt
~/vampyre/
  app/
  artifacts/
  config/
    project-registry.json
    vampyre.env
  data/
    vampyre.sqlite
  logs/
  repos/
  reports/
  worktrees/
```

`vampyre.env` is created on the host by `host setup` with `0600` permissions. Required secret keys checked by the current doctor path are `GITHUB_TOKEN`, `TELEGRAM_BOT_TOKEN`, and `TELEGRAM_CHAT_ID`. `OPENROUTER_API_KEY` is treated as optional by the doctor.

## CLI Surface

The CLI entrypoint is `src/cli.ts`; the package bin points to `dist/cli.js`.

Implemented command groups:

- `doctor` - checks host readiness without printing secret values.
- `host setup` - creates runtime directories and the env stub, then checks Node and pnpm.
- `daemon run` - runs the foreground daemon loop.
- `daemon install|start|stop|restart|status|logs` - deploys built artifacts and wraps `systemd --user`.
- `status` - renders the Owner check-in from runtime state.
- `pause 1m|1h|1d`, `pause status`, `resume` - manages global Work Pause state.
- `github check` - checks GitHub auth and optional repository access.
- `approval check` - checks GitHub approval issues for the `VAMPYRE_APPROVED` marker.
- `review request` - creates or reuses a GitHub review issue and sends a Telegram link.
- `pr upsert` - creates or updates a GitHub pull request and sends a Telegram link.
- `watcher discover` - inspects a managed Safe/Watcher repository and writes reports.
- `validation request` - dispatches and optionally waits for a configured external native-validation workflow.
- `agent run` - runs the Worktree Build Agent loop.
- `builder repo create` - creates or confirms an approved private Builder repository and writes initial project files.
- `ping telegram` and `-ping telegram` - sends a Telegram test message.

## Source Modules

- `src/remote/` validates and expands workspace paths for local and SSH command contexts.
- `src/doctor/` runs SSH and host readiness checks.
- `src/host/` creates the runtime workspace and env stub.
- `src/daemon/` builds the service unit, manages `systemd --user`, runs heartbeat ticks, polls Telegram commands, runs scheduler/control-surface steps, and launches the Build Agent when selected.
- `src/registry/` loads the Project Registry and creates a default registry if missing.
- `src/state/` owns SQLite migrations and runtime state access.
- `src/scheduler/` plans budget-aware project selection, cadence deferral, Work Pause deferral, and the single Active Build Agent limit.
- `src/budget/` reads Codex JSONL usage from `CODEX_HOME` or `~/.codex`.
- `src/checkin/` builds CLI and Telegram check-in summaries from Operational State.
- `src/control/` manages Work Pause commands.
- `src/telegram/` polls Telegram updates for authorized `/status`, `/pause1min`, `/pause1hour`, `/pause1day`, and `/resume`.
- `src/github/` wraps GitHub checks, approval lookup, review issue workflow, PR upsert workflow, and shared API helpers.
- `src/validation/` dispatches GitHub Actions native-validation workflows, persists external run state, and writes validation reports.
- `src/watcher/` runs Safe/Watcher discovery against managed project clones.
- `src/agent/` runs validation, worker task context, worker command execution, worktree output, GitHub review/PR output, Telegram notification, reports, and blocker handling.
- `src/builder/` handles approved Builder repository creation for the `pinmark` template.
- `src/ping/` sends Telegram test messages.
- `src/status/` renders the status/check-in command.

## SQLite State

Migrations are embedded in `src/state/operationalState.ts`.

Current tables created by migrations:

- `schema_migrations`
- `projects`
- `run_journals`
- `project_blockers`
- `idempotency_keys`
- `scheduler_cursors`
- `scheduler_ticks`
- `active_build_agent_lock`
- `work_pause`
- `telegram_update_cursor`
- `notification_delivery_state`
- `telegram_unauthorized_attempt_state`
- `external_validation_runs`

Operational State syncs Project Registry profiles into SQLite on startup/status reads. Scheduler ticks persist the latest tick plus per-project cursors. Work Pause and Telegram update cursor state are also persisted in SQLite.

## Project Registry

If `~/vampyre/config/project-registry.json` is missing, `src/registry/projectRegistry.ts` creates version `1` with two default profiles:

- `palette-wow`: Safe/Watcher Mode for `scwlkr/paletteWOW`, with Rails validation commands.
- `screenshot-tool`: Builder Mode for `scwlkr/pinmark`, with `git diff --check` validation and direct-main product-loop autonomy.

Registry validation rejects unsupported modes, duplicate ids, and missing mode-specific fields.

## Runtime Flow

1. `daemon install` requires built `dist/` artifacts, copies `dist` and `package.json` to `~/vampyre/app`, writes `~/.config/systemd/user/vampyre.service`, reloads systemd, and enables the service.
2. `daemon run` initializes Operational State from the runtime workspace.
3. Each daemon heartbeat refreshes state, polls Telegram operational commands, records a scheduler tick, runs the review control surface when a project is selected, and invokes the Build Agent if an eligible project remains selected.
4. The Build Agent fetches the managed repo, creates a disposable worktree, runs validation from the registry or watcher discovery report, optionally runs a worker command, records a Run Journal, writes reports, and either pushes output or records blockers.
5. GitHub remains the durable review/approval surface. Telegram sends notifications and low-risk command replies.

## Gaps

- No CI configuration is present.
- No `.env.example` is present; host setup creates the runtime env stub instead.
- Build Agent adoption of native validation is still pending; the first version is operator-triggered through `validation request`.
